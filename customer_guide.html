Building a Subaccount Call & Conference Log Dashboard

Using Twilio Event Streams to export read-only data at the subaccount level

1. The Problem

Twilio Console does not currently support granting read-only access to call or conference logs at the subaccount level. Any user with Console access to a subaccount has full administrative rights — there is no viewer-only role.

This creates a real gap for customers in situations like:

  • A managed service provider with one subaccount per client who wants to give each client visibility into their own call data.
  • An internal team that needs to audit call volume and error rates without access to account settings.
  • A compliance officer who needs a read-only log of all calls across subaccounts.
Twilio Support's recommendation: Use Event Streams to export Voice Insights and Conference Insights events in real time to a webhook, then build a custom read-only visualization layer on top of that data.

2. Solution Overview: Twilio Event Streams

Twilio Event Streams is a first-party Twilio product that pushes real-time events from your account to a destination of your choice — a webhook, Amazon Kinesis, or Segment. For this use case, a webhook is the simplest approach.

Every time a call or conference completes on your account or any subaccount, Twilio's Voice Insights engine generates a rich event containing call details, quality metrics, and — critically — the account_sid field that identifies which subaccount the call originated from.

Twilio Master Account
├── Subaccount A (ACaaa...)
├── Subaccount B (ACbbb...)
└── Subaccount C (ACccc...)
        │
        │ Voice Insights events (~60s after call ends)
        ▼
Event Streams Subscription (on master account)
        │
        │ HTTP POST — CloudEvents 1.0 format
        ▼
Your Webhook → Database → Read-Only Dashboard
                          (filtered by account_sid)

By creating the subscription on the master account, you receive events from all subaccounts in one stream. The account_sid in each event tells you which subaccount it belongs to, enabling per-subaccount filtering in your dashboard.

3. Relevant Event Types

Event Type Fires When
com.twilio.voice.insights.call-summary.complete A call completes and full Voice Insights processing finishes
com.twilio.voice.insights.call-summary.partial Partial summary available (~30s after call ends, faster)
com.twilio.voice.insights.conference-summary.complete A conference ends
com.twilio.voice.insights.conference-participant-summary.complete A conference participant disconnects

4. Event Payload Structure

Twilio Event Streams uses the CloudEvents 1.0 specification. Each HTTP POST contains a JSON array. Call data lives inside the data object and uses snake_case field names. 

[
  {
    "specversion": "1.0",
    "type": "com.twilio.voice.insights.call-summary.complete",
    "id": "EZxxxxxxxxxxxxxxxxxxxxxxxxxxxxx",
    "time": "2026-04-23T02:08:35.000Z",
    "data": {
      "call_sid":       "CAxxxxxxxxxxxxxxxxxxxxxxxxxxxxx",
      "account_sid":    "ACxxxxxxxxxxxxxxxxxxxxxxxxxxxxx",  // subaccount identifier
      "call_state":     "completed",
      "duration":       45,
      "start_time":     "2026-04-23T02:07:50Z",
      "end_time":       "2026-04-23T02:08:35Z",
      "properties": {
        "direction":    "outbound_api"
      },
      "from": { "caller": "+15551234567" },
      "to":   { "callee": "+15559876543" }
    }
  }
]

Key fields to store:

  • data.account_sid — which subaccount the call belongs to
  • data.call_sid — the Call SID for cross-referencing with the REST API
  • data.call_state — final status (completed, failed, busy, no-answer)
  • data.duration — call duration in seconds
  • data.properties.direction — inbound, outbound_api, outbound_dial
  • data.from.caller / data.to.callee — phone numbers

5. Step-by-Step Implementation

Step 1 — Prerequisites

  • A Twilio account with one or more subaccounts
  • Voice Insights enabled (available on all accounts by default)
  • A public HTTPS URL for your webhook (use ngrok for local development)
  • Python 3.9+ with the twilio library installed: pip install twilio flask

Step 2 — Create a Sink (the webhook destination)

Run this once using your master account credentials:

from twilio.rest import Client

client = Client("ACyour_master_account_sid", "your_master_auth_token")

sink = client.events.v1.sinks.create(
    description="Call & Conference Log Dashboard",
    sink_configuration={
        "destination": "https://your-app.com/webhook/events",
        "method": "POST",
        "batch_events": False,
    },
    sink_type="webhook",
)
print("Sink SID:", sink.sid)  # Save this — DGxxxxx

Step 3 — Create a Subscription

subscription = client.events.v1.subscriptions.create(
    description="Voice Insights — Subaccount Dashboard",
    sink_sid=sink.sid,
    types=[
        {"type": "com.twilio.voice.insights.call-summary.complete"},
        {"type": "com.twilio.voice.insights.call-summary.partial"},
        {"type": "com.twilio.voice.insights.conference-summary.complete"},
        {"type": "com.twilio.voice.insights.conference-participant-summary.complete"},
    ],
)
print("Subscription SID:", subscription.sid)

Step 4 — Build the Webhook Receiver

Your webhook must validate the Twilio signature on every request, then parse the CloudEvents array and store each event in your database. 

from flask import Flask, request, jsonify
from twilio.request_validator import RequestValidator

app = Flask(__name__)
MASTER_AUTH_TOKEN = "your_master_auth_token"

@app.post("/webhook/events")
def receive_event():
    # 1. Validate Twilio signature
    validator = RequestValidator(MASTER_AUTH_TOKEN)
    sig  = request.headers.get("X-Twilio-Signature", "")
    body = request.get_data(as_text=True)
    if not validator.validate(request.url, body, sig):
        return jsonify({"error": "Forbidden"}), 403

    # 2. Parse the CloudEvents array
    for event in request.get_json(force=True) or []:
        event_type = event.get("type")
        data       = event.get("data", {})

        account_sid = data.get("account_sid")       # subaccount identifier
        call_sid    = data.get("call_sid")
        status      = data.get("call_state")
        duration    = data.get("duration")
        direction   = (data.get("properties") or {}).get("direction")
        from_number = (data.get("from") or {}).get("caller")
        to_number   = (data.get("to")   or {}).get("callee")
        started_at  = data.get("start_time")

        # 3. Store in your database (insert or upsert on call_sid)
        save_to_db(
            account_sid=account_sid,
            call_sid=call_sid,
            status=status,
            duration=duration,
            direction=direction,
            from_number=from_number,
            to_number=to_number,
            started_at=started_at,
            raw_payload=event,        # always store the full event
        )

    return jsonify({"status": "ok"}), 200

Step 5 — Database Schema

Store events in a relational database (PostgreSQL recommended for production, SQLite for development). Index on account_sid and started_at for fast subaccount filtering.

-- Call logs
CREATE TABLE call_logs (
    id               SERIAL PRIMARY KEY,
    account_sid      VARCHAR(34) NOT NULL,
    call_sid         VARCHAR(34) UNIQUE NOT NULL,
    status           VARCHAR(20),
    direction        VARCHAR(20),
    from_number      VARCHAR(20),
    to_number        VARCHAR(20),
    duration_seconds INTEGER,
    started_at       TIMESTAMP,
    ended_at         TIMESTAMP,
    raw_payload      JSONB        -- full original event for replay
);

CREATE INDEX idx_call_logs_account_sid ON call_logs (account_sid);
CREATE INDEX idx_call_logs_started_at  ON call_logs (started_at);

-- Conference logs
CREATE TABLE conference_logs (
    id               SERIAL PRIMARY KEY,
    account_sid      VARCHAR(34) NOT NULL,
    conference_sid   VARCHAR(34) UNIQUE NOT NULL,
    friendly_name    VARCHAR(255),
    status           VARCHAR(20),
    participant_count INTEGER,
    duration_seconds INTEGER,
    started_at       TIMESTAMP,
    raw_payload      JSONB
);

CREATE INDEX idx_conference_logs_account_sid ON conference_logs (account_sid);
Why store raw_payload? Twilio occasionally updates Voice Insights event schemas. Storing the full original event means you never lose data — you can re-parse it at any time if your schema needs to change.

Step 6 — Build the Dashboard

The dashboard queries your database filtered by account_sid. Because this is your own application, you control access completely.

Recommended features:

  • Subaccount filter — dropdown of distinct account_sid values
  • Date range filter — filter on started_at
  • Status filter — completed, failed, busy, no-answer
  • Call volume chart — daily counts grouped by subaccount
  • Duration chart — average call duration per subaccount
  • Error rate chart — percentage of failed/busy/no-answer calls over time
  • Call log table — paginated, sortable list of all calls

6. Access Control

Because the dashboard is your own application, you have full control over who sees what. A simple approach for giving clients read-only access to their own subaccount:

  1. Each client user is assigned their subaccount's account_sid at login.
  2. Every database query is scoped to that account_sid — users can never see other subaccounts' data.
  3. The application layer has no Twilio credentials — it only reads from the database.
  4. Clients cannot modify any Twilio configuration because they never touch Twilio directly.

7. Best Practices

Handle schema changes gracefully

Always store the full raw event payload in a JSON column. Access all fields with safe fallbacks (.get() in Python). When Twilio updates an event schema, re-parse from stored raw payloads without data loss.

Monitor for dropped events

  • Store the X-Twilio-Event-Id header from each request. Gaps in sequential IDs indicate dropped events.
  • Maintain a dead-letter queue — any event that fails to parse should be flagged for retry.
  • Expose a /health endpoint showing the timestamp of the last received event and count of unprocessed events.
  • Periodically cross-reference against the Twilio REST API: client.calls.list() per subaccount vs. database row counts.

Signature validation

  • Never disable Twilio signature validation in production.
  • If running behind a reverse proxy (nginx, load balancer), ensure request.url matches what Twilio signed. Use ProxyFix middleware in Flask.
  • Use the Auth Token of the account that owns the subscription — master account token for master-level subscriptions.

Event timing

  • call-summary.partial arrives ~30 seconds after call completion — good for near-real-time display.
  • call-summary.complete arrives within a few minutes — includes full quality metrics.
  • Upsert on call_sid so partial and complete events update the same row.

8. Optional: Grafana Integration

If the customer wants advanced visualization without building a custom frontend, Grafana connects directly to PostgreSQL and provides rich charting out of the box.

  1. Run Grafana: docker run -p 3000:3000 grafana/grafana
  2. Add a PostgreSQL data source pointing to your database.
  3. Build panels using direct SQL queries:
-- Call volume by subaccount over time
SELECT
  date_trunc('day', started_at) AS time,
  account_sid,
  count(*)                       AS calls
FROM call_logs
WHERE started_at BETWEEN $__timeFrom() AND $__timeTo()
GROUP BY 1, 2
ORDER BY 1;

Use Grafana template variables to create a subaccount dropdown that filters all panels simultaneously.

9. Frequently Asked Questions

QuestionAnswer
Does Event Streams cost extra? Event Streams is free to set up. Voice Insights Advanced features have associated costs — check current pricing. Basic call summaries have minimal cost impact.
Does this work for inbound calls? Yes. The direction field distinguishes inbound, outbound_api, and outbound_dial.
What if my webhook is down? Twilio retries failed webhook deliveries automatically. Store event IDs to detect any permanent gaps.
Can I subscribe per subaccount instead? Yes — create a separate subscription on each subaccount using that subaccount's credentials. More granular but more overhead. Master-account subscription is simpler for most use cases.
How far back does data go? Event Streams only delivers events going forward from when the subscription is created. Historical data must be fetched via the REST API separately.
Can I use this for SMS logs too? Yes — Event Streams supports messaging events (com.twilio.messaging.message.*). The same pattern applies.
